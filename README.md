# MKTd02 — Cryptographically Verifiable Deletion Receipts for ICP

A composable Rust library that any Internet Computer canister can import to produce **CVDRs** (Cryptographically Verifiable Deletion Receipts) for GDPR right-to-erasure compliance.

**Configuration:** MKTd | Library | ICP (StableCell / StableBTreeMap) | Single-canister | Leaf mode | Tombstone (Explicit) | ICP subnet BLS certification

## What MKTd02 Proves

A deletion receipt generated by MKTd02 proves:

1. The canister's PII-bearing state had hash H1 (`pre_state_hash`) before deletion.
2. All PII fields were replaced with the published tombstone constant, producing state hash H2 (`post_state_hash`).
3. This transition is bound to a specific point in time via `deletion_event_hash` (which includes timestamp, module hash, manifest hash, and nonce).
4. H2 and the deletion event are certified by the hosting subnet's threshold BLS signature via `certified_commitment`.
5. The canister was running code with module hash M at the time of deletion.

**What MKTd02 does not prove:**

- That the deleted fields contained PII (MKTd02 operates on bytes, not semantics).
- That no copies exist in other canisters, snapshots, backups, or off-chain systems.
- That the tombstone will not be reversed by a future canister upgrade (resurrection is *detectable*, not *prevented*).
- That the adapter faithfully represents the complete PII state (depends on correct adapter implementation).

```toml
[dependencies]

hex = "0.4"
```

### 2. Implement the adapter trait

```rust
impl MKTdDataSource for ProfileAdapter {
    fn mode(&self) -> CommitMode { CommitMode::Leaf }
    fn pii_field_manifest(&self) -> Vec<FieldDescriptor> { /* ordered PII fields */ }
    fn manifest_hash(&self) -> [u8; 32] { compute_manifest_hash(&self.pii_field_manifest()) }
    fn get_state_bytes(&self) -> Vec<u8> { encode_pii_state(&my_pii).unwrap() }
    fn tombstone_state(&mut self) { /* write hex-encoded TOMBSTONE_CONSTANT to all PII fields */ }
    fn is_tombstoned(&self) -> bool { /* check all PII fields == hex-encoded TOMBSTONE_CONSTANT */ }
}
```

### 3. Wire into canister lifecycle

```rust
#[init]
fn init(owner: Principal) {
    // ... your init logic ...
    let adapter = ProfileAdapter;
    let module_hash = [0u8; 32]; // zeros for dev; real post-shrink WASM hash for production
    MEMORY_MANAGER.with(|mm| {
        mktd02::init(&adapter, &mm.borrow(), MktdConfig {
            base_memory_id: 100,
            subnet_id: Principal::anonymous(), // real subnet ID for production
        }, module_hash);
    });
}
```

This makes init() consistent with post_upgrade() — both now receive module_hash, matching the actual committed library API.

```rust
fn delete_profile() -> Result<String, MyError> {
    let mut adapter = ProfileAdapter;

    Ok(hex::encode(receipt_id))
}
```

## CVDR Receipt Structure

A Leaf-mode receipt contains the following fields:

| Field | Type | Description |
|---|---|---|
| `receipt_id` | `[u8; 32]` | SHA-256(MKTD02_RECEIPT_V1 ‖ canister_id ‖ timestamp ‖ nonce) |
| `canister_id` | `Principal` | Principal of the canister that executed the deletion |
| `subnet_id` | `Principal` | Subnet hosting the canister |
| `commit_mode` | `String` | `"Leaf"` — single data subject per canister |
| `pre_state_hash` | `[u8; 32]` | Hash of PII state before deletion |
| `post_state_hash` | `[u8; 32]` | Hash of PII state after tombstoning |
| `tombstone_hash` | `[u8; 32]` | Domain-separated hash binding canister, tombstone constant, timestamp, nonce |
| `deletion_event_hash` | `[u8; 32]` | Domain-separated hash binding pre/post state, timestamp, module, manifest, nonce |
| `certified_commitment` | `[u8; 32]` | Domain-separated hash binding post_state_hash + deletion_event_hash |
| `manifest_hash` | `[u8; 32]` | Hash of the PII field manifest at time of deletion |
| `module_hash` | `[u8; 32]` | Canister WASM module hash at time of deletion |
| `timestamp` | `u64` | ICP system time in nanoseconds |
| `nonce` | `u64` | Monotonic counter (never reused, even across upgrades) |

**Representation:** The receipt is an unsigned artifact. Receipts are stored as deterministic CBOR in stable memory and returned as Candid structs via `mktd_get_receipt()`. Verification relies on the certified commitment (published via `set_certified_data()`) and the subnet's BLS signature, not a signature on the receipt itself.

## Cryptographic Specification

All hash operations use SHA-256. Domain separation is enforced via `DomainTag` newtype and `hash_with_tag()`, which prepends the tag bytes before the payload.

### Hash Formulas

```
TOMBSTONE_CONSTANT = SHA-256("MKTD_TOMBSTONE_V1")
    — 32-byte value written to each PII field during tombstoning

mktd_salt = SHA-256("MKTD02_SALT_V1" ‖ canister_id_bytes)
    — derived at runtime, NOT stored in stable memory
    — anti-correlation measure (prevents cross-canister linkage), not a secret

state_hash = SHA-256(encode_pii_state(pii_fields))
    — deterministic CBOR encoding of all PII fields per manifest order

tombstone_hash = SHA-256("MKTD02_TOMBSTONE_HASH_V1" ‖ canister_id ‖ TOMBSTONE_CONSTANT ‖ timestamp ‖ nonce)

deletion_event_hash = SHA-256("MKTD02_EVENT_V1" ‖ pre_state_hash ‖ post_state_hash ‖ timestamp ‖ module_hash ‖ manifest_hash ‖ nonce)

certified_commitment = SHA-256("MKTD02_CERTIFIED_V1" ‖ post_state_hash ‖ deletion_event_hash)
    → published via ic0.certified_data_set()

receipt_id = SHA-256("MKTD02_RECEIPT_V1" ‖ canister_id ‖ timestamp ‖ nonce)

manifest_hash = SHA-256("MKTD02_MANIFEST_V1" ‖ field_descriptors_cbor)
```

### Naming Convention Table

| Name | Kind | Purpose |
|---|---|---|
| `MKTD_TOMBSTONE_V1` | Constant seed | SHA-256 seed → `TOMBSTONE_CONSTANT` (the value written to PII fields) |
| `MKTD02_TOMBSTONE_HASH_V1` | Domain tag | Prefix for `tombstone_hash` computation |
| `MKTD02_EVENT_V1` | Domain tag | Prefix for `deletion_event_hash` computation |
| `MKTD02_CERTIFIED_V1` | Domain tag | Prefix for `certified_commitment` computation |
| `MKTD02_RECEIPT_V1` | Domain tag | Prefix for `receipt_id` derivation |
| `MKTD02_SALT_V1` | Domain tag | Prefix for per-canister `mktd_salt` derivation |
| `MKTD02_MANIFEST_V1` | Domain tag | Prefix for `manifest_hash` computation |

**Key distinction:** The tombstone constant is a *value written to storage*; domain tags are *prefixes for hash computations*. These must not be confused.

## Verification Procedures

### V1: State Transition Verification

Verifier obtains the receipt. Using the published cryptographic spec:

1. Confirm `pre_state_hash` ≠ `post_state_hash` (state actually changed).
2. Recompute `tombstone_hash` from receipt fields + published `TOMBSTONE_CONSTANT`. Confirm match.
3. Recompute `deletion_event_hash` from `pre_state_hash`, `post_state_hash`, `timestamp`, `module_hash`, `manifest_hash`, `nonce`. Confirm match.
4. Recompute `certified_commitment` from `post_state_hash` + `deletion_event_hash`. Confirm match.

**Result:** The state transition is internally consistent. All hash fields are deterministically reproducible from the receipt contents and the published cryptographic spec.

### V2: Subnet Certificate Verification

Verifier obtains a certified query response from the canister (or uses the certificate embedded in an exported receipt):

1. Verify the BLS certificate chain: subnet signature → subnet public key → NNS root of trust.
2. Extract certified data from the certificate.
3. Confirm certified data matches `certified_commitment` in the receipt.

**Result:** The canister's certified state includes the post-deletion state hash and the deletion event. The subnet (2/3 of nodes) attests to this state.

### V3: Module Hash Verification

1. Query the canister's current module hash via `canister_status` on ICP's management canister.
2. Compare against `module_hash` in the receipt.
3. Compare against the published WASM hash (from deterministic build).

**Result:** If current module hash matches the receipt's hash, the canister code has not changed since deletion. If different, the canister has been upgraded — but the receipt remains valid evidence of what occurred under the prior code version.

### V4: Tombstone Persistence Check (optional, point-in-time)

1. Call `mktd_get_tombstone_status()` on the canister.
2. Confirm `is_tombstoned` is true and `tombstoned_at` is set.
3. Call `mktd_get_state_hash()` and confirm the current state hash matches `post_state_hash` in the receipt.

**Result:** The tombstone has not been reversed since the receipt was generated. This is a point-in-time check, not a continuous guarantee. If the tombstone has been reversed, the state hash will differ.

## Verification reference code

Independent verification tools for CVDRs produced by this library are maintained in a separate repository: [CVDR-Verify](https://github.com/Together-Alone-Ventures/CVDR-Verify).

The verification tooling implements procedures V1–V4:
- **V1** — Cryptographic hash recomputation (receipt integrity)
- **V2** — BLS subnet certificate verification (ICP trust anchor)
- **V3** — Module hash comparison (code provenance)
- **V4** — Tombstone persistence check (deletion durability)

## Stable Memory Coordination

MKTd02 allocates **8 stable memory slots** starting from a configurable base MemoryId (default: 100).

| Offset | Content | Type | Notes |
|---|---|---|---|
| base+0 | Meta cell | u32, u32, Option\<u64\>, [u8;32] | schema_version, memory_base, initialised_at, module_hash |
| base+1 | state_hash | [u8; 32] | Current PII state hash |
| base+2 | nonce | u64 | Monotonic, never reused |
| base+3 | certified_commitment | [u8; 32] | Published via set_certified_data() |
| base+4 | deletion_event_hash | [u8; 32] | Initially zero hash. Set during deletion |
| base+5 | manifest_hash | [u8; 32] | Frozen PII boundary. Change triggers upgrade cascade |
| base+6 | receipt store | StableBTreeMap<[u8;32], Vec\<u8\>> | Keyed by receipt_id. Max 8192 bytes per receipt |
| base+7 | tombstoned_at | Option\<u64\> | Engine-owned. Only execute_deletion() writes this |

**Range check:** `base + 7` must be `<= 255`, validated on init.

**Collision detection:** If meta cell has `schema_version` ≠ 0, `memory_base` ≠ 0, or `initialised_at` set, and `memory_base` differs from requested base, the library traps with a descriptive error.

**Salt:** The `mktd_salt` used in hash computations is NOT stored in stable memory. It is deterministically derived at runtime: `SHA-256("MKTD02_SALT_V1" ‖ canister_id_bytes)`.

Your canister must not allocate stable structures in the range `[base, base+7]`.

## Guard Checklist

Three options for protecting PII-mutating functions:

| Approach | Signature | Behaviour | When to use |
|---|---|---|---|
| Guard function (`mktd_guard_check()`) | Returns `Result<(), E>` | Returns `Err` on violation | Error type in separate crate (orphan rule) |
| `#[mktd_guard]` macro | Returns `Result<T, E: GuardError>` | Returns `Err` on violation | Error type in same crate |
| `mktd02::assert_can_write()` | `()` | Traps on violation | Non-Result functions |

All three check: (1) MKTd02 is initialised, (2) canister is not tombstoned.

**Two distinct tombstone checks exist and must not be confused:**
- `mktd02::is_tombstoned()` reads the engine-owned `tombstoned_at` timestamp — this is the **write guard**.
- `adapter.is_tombstoned()` checks PII field values — this is the **post-condition check** during deletion.

## Upgrade Behaviour

On every `post_upgrade`:

1. **Manifest check:** If `manifest_hash` changed → full recomputation cascade (state_hash → certified_commitment → publish).
2. **Module hash:** Updated unconditionally in meta cell.

Old receipts remain verifiable under the `manifest_hash` recorded in each receipt.

**Endianness:** Stable memory encoding uses little-endian; hash preimages use big-endian. These are separate domains.

## Module Hash Pipeline Rule

> **Hash what you ship.**

The module hash embedded in the receipt must be computed from the **final post-ic-wasm-shrink WASM bytes** — the exact bytes that get deployed. For local dev, zeros are accepted. For production: two-pass build or pass hash via factory's `install_code` argument. Module hash is captured at init/upgrade time, not per-deletion.

## Deterministic Encoding Checklist

For adapters implementing `MKTdDataSource`:

- [ ] PII fields listed in `pii_field_manifest()` in ascending `field_order`
- [ ] `get_state_bytes()` uses `encode_pii_state()` (not raw ciborium)
- [ ] `tombstone_state()` writes hex-encoded `TOMBSTONE_CONSTANT` to **every** PII field
- [ ] `is_tombstoned()` checks **all** PII fields == hex-encoded `TOMBSTONE_CONSTANT`
- [ ] No `f32`/`f64` fields in PII state (floats are rejected by `encode_pii_state()`)
- [ ] No `HashMap`/`BTreeMap` at the PII struct level (use structs with named fields)
- [ ] Field order in manifest matches field order in serialisation struct
- [ ] CBOR encoding is deterministic under `encode_pii_state()`; this is deterministic CBOR under this library's encoder (ciborium + serde), **not** RFC 8949 canonical CBOR

## Residual Trust Statement

| # | Assumption | Mitigation |
|---|---|---|
| RT1 | ICP subnet consensus is honest (2/3 of nodes not colluding) | Foundation of ICP security model. Subnet membership governed by NNS. Sovereign subnets / UTOPIA change this — operator controls all nodes. |
| RT2 | Canister controller does not upgrade canister to reverse tombstones | Module hash verification (V3) detects code changes. DAO/governance control of upgrades mitigates. Receipts with prior module hash remain valid evidence of what happened under that code version. |
| RT3 | MKTdDataSource adapter correctly maps all PII fields | Open-source adapter code is auditable. Manifest hash freezes the PII boundary at deletion time. |
| RT4 | Stable memory overwrite is durable | ICP stable memory overwrites logically. Freed regions may retain byte patterns until reused. MKTd02 operates at the logical level — physical byte scrubbing is outside scope. |
| RT5 | No canister snapshots contain pre-deletion state | Snapshot restoration would change the certified commitment, detectable by verifiers. Snapshot lifecycle management is the controller's operational responsibility. |
